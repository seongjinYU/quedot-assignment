## 1. 네이버 봇 차단 돌파 — 단순 HTTP → 인증 세션 재사용

**고민한 지점**
- `curl`/`fetch` 단순 요청은 네이버에서 즉시 429(분당 요청 제한 + JS 동적 렌더).
- Playwright 기본 브라우저도 자동화 탐지(`navigator.webdriver`)로 캡차/로그인 벽.

**트레이드오프**
- 차단 우회 기술(프록시 로테이션, 캡차 솔버)을 쓸 수도 있었으나 — 운영 리스크·합법성 문제.
- 대신 **사용자가 1회 직접 로그인한 인증 세션을 persistent context로 재사용** + stealth.
  - 장점: 안정적, 합법적(본인 계정), 캡차 없음. 단점: 최초 1회 수동 로그인 필요.

**결론**: "차단을 뚫는 기술력"보다 "리스크를 의식한 정공법"을 택함. 실제 서비스라면 공식 API·제휴를 우선 검토.

## 2. 결정적 추출 — 화면 파싱 대신 내부 JSON API

**고민한 지점**
- HTML 셀렉터 파싱은 사이트 구조 변경에 취약(과제의 "결정적 추출" 요구와 배치).

**선택**
- 네이버: 화면이 호출하는 **내부 JSON API**(`/i/v2/...`, `/n/v2/...`)를 세션 컨텍스트에서 직접 호출.
  - 상세(이름·정가·옵션·이미지·카테고리), 가격(product-benefits), 목록(페이지네이션) 모두 정형 JSON.
- happyland(고도몰): 목록 카드 `data-*` 속성 + `layer_option.php`(AJAX) → cheerio 파싱.

**트레이드오프**
- 내부 API는 비공식이라 네이버가 예고 없이 바꿀 수 있음(경로 하드코딩 위험) → RECON_NOTES에 정찰 근거 기록, 변경 시 재정찰 전제.
- 목록 페이지네이션은 직접 fetch가 429(앱 서명헤더 필요) → **페이지 버튼 클릭 후 XHR 가로채기**로 우회.

## 3. 어댑터 패턴 — smartstore/brand/happyland 통합

**고민한 지점**: "다른 링크를 넣어도 작동" 요구 + 3개 스토어 구조가 제각각.

**선택**: `StoreAdapter` 인터페이스 + 도메인 기반 자동 선택.
- 네이버 2종(smartstore=`/i/`, brand=`/n/`)은 prefix 차이만으로 **하나의 어댑터**가 처리.
- happyland는 완전히 다른 구조(고도몰)라 별도 어댑터. `needsBrowser=false`로 브라우저 없이 순수 HTTP.

**효과**: 새 몰 = 어댑터 1개 추가. 네이버는 brand 추가 시 코드 거의 재사용.

## 4. 성능 — 목록 우선 + 가격 배치 + 호출 최소화

**고민한 지점**: 상품 50~200개를 하나씩 상세 진입하면 느림(딜레이 포함 10분+).

**선택**
- 목록 1회로 얻는 필드는 상품마다 재호출 안 함.
- 네이버 가격은 `product-benefits`에 **여러 상품을 배치**로 한 번에(개별 호출 회피).
- happyland는 브라우저 렌더 없이 **순수 HTTP fetch** + 옵션 있는 상품만 옵션 호출.
- 딜레이(rate limit)는 **줄이지 않음** — 차단 위험. 속도는 "호출 횟수 감소"로 확보.

**트레이드오프**: 옵션·USP는 상품별 처리가 불가피 → 전수 시 시간 소요. 증분 재크롤(가산)은 미구현.

## 5. AI 활용 — 룰 우선, LLM은 비정형만 + 환각 차단

**고민한 지점**: 명세가 "LLM으로 옵션 정규화/분류/USP" + "AI 출력 신뢰하지 말 것" 동시 요구.

**선택 (OpenAI structured output)**
- 정형 데이터(가격·이미지·이름)는 코드로 결정적 처리. LLM은 **category/hashtags/usp/옵션 의미론 정규화**만.
- structured output(json_schema)으로 출력 강제, category는 **7종 enum**으로 제약.
- **환각 차단을 validate.ts 단일 관문으로 통합** (provider/룰/fallback 경로 무관하게 최종 통과).
- LLM 실패·개수 불일치 시 **룰 baseline fallback** → 키 없어도 동작.

**핵심 교훈 (실수 → 수정)**
- 초기엔 환각 가드가 provider에 흩어져 있어, 근거 없는 happyland 상품에서:
  - 모자 → "기타 식품" 오분류, USP 환각, **provenance가 "categoryPath 기반"이라 거짓 표기**(실제 null).
- → 가드를 validate로 단일화 + `meta.basis`(근거 유무)를 정직하게 기록.
  - 근거 없으면 USP=null, 다중분류 축소, provenance에 실제 근거만 표기.

## 6. 상세 본문 USP — "텍스트 추출"의 환상과 현실 ★ (가장 솔직한 회고)

**고민한 지점**: USP는 상세설명이 가장 좋은 근거. 상세 본문 텍스트를 추출하려 함.

**검증 과정에서 드러난 현실 (3단계로 깨달음)**
1. 네이버 상세는 **거의 100% 이미지**(textContent 0자, img 45장 등). 한국 이커머스의 구조적 특성.
2. 별도 contents API를 호출해 긁어보니, 건지는 건 본문 문장이 아니라 **SEO 키워드 더미**
   ("칼슘 뼈영양제 골다공증영양제...", "감마리놀렌산 여성갱년기 갱년기영양제...").
3. 그 키워드는 사실 **별도 호출도 필요 없이** 상세 API 응답의 `detailContents.detailContentText`에
   이미 들어있었음 — 즉 별도 호출은 순수 낭비였고, 그 필드 자체가 **셀러태그와 중복되는 SEO 키워드**였음.

→ 초기엔 이를 "상세 본문 추출 성공(detailText=true)"이라 **두 번이나 과장 표기**했다가,
   실제 데이터를 눈으로 검증한 뒤 두 번 다 정정함. (검증 없이 단정한 것이 실수)

**트레이드오프 / 최종 결정**
- 별도 contents 호출 → **제거** (상품당 +1 요청 낭비, 건지는 게 키워드뿐).
- `detailContents.detailContentText`(SEO 키워드 필드) → **USP 근거에서 의도적으로 제외**.
  - 이유: 문장형 본문이 아니라 키워드 나열이고, 이미 `seoInfo.sellerTags`(hashtags 근거)와 중복.
  - "상세 본문 추출"이라 부르면 과장 → 정직하게 빼는 것이 provenance 신뢰성에 부합.
- USP 근거는 **상품명 + 셀러태그 + 카테고리**로 단일화(전 상품 `detailText=false`로 통일, 검증 완료).
- **방어 로직은 유지**: 다른 브랜드몰이 진짜 문장형 상세(`detailContent` 단수 등)를 줄 경우 자동 사용.
  추가 네트워크 호출 없이 상세 응답에 이미 있는 문장 본문만(30자+) 채택.

**핵심 교훈**
> "상세 본문 텍스트 추출"처럼 그럴듯한 기능도 실제 데이터로 검증하니 허상이었다.
> 그럴듯한 기능을 남기는 것보다, 검증으로 허상을 걷어내고 근거를 정직하게 표기하는 것이 더 가치 있었다.

**해결 못한 것 / 더 시간이 있었다면**
- 상세 이미지 OCR/Vision으로 진짜 본문을 읽으면 USP 품질↑ (구현 복잡·비용·시간 문제로 보류).
  - **(후속 검증, 2026-06-10) happyland OCR 도입을 재검토 → 실물 검증 후 "도입 안 함" 확정.**
    - 가설: "모든 상품 상세 상단에 한 줄 설명 이미지가 일정하게 있다" → OCR로 USP 근거 확보.
    - 정찰(`recon-happyland.ts`, 3상품): 상세영역 `.cont_detail`·첫 이미지 730×730 **위치·규격은 일관**(가설처럼 보임).
    - 그러나 **이미지 요소를 직접 스크린샷해 실물 확인**하니, 첫 이미지도 긴 [1] 이미지(1500×1980)도 **전부 텍스트 없는 상품 사진**이었음. `alt`도 상품명뿐.
    - → happyland 상세엔 OCR로 뽑을 설명 텍스트 자체가 없음. 도입해도 USP 근거 0, 비용·복잡도만 증가 → **구현 전 검증으로 차단**.
  - **교훈(=#6 재확인)**: "패턴이 일관적"(위치·규격)이라는 신호만으로 "설명 이미지가 있다"고 단정하면 또 허상이었다. 위치 패턴 ≠ 내용. 실물 픽셀을 봐야 안다.
  - **★ 정정 (2026-06-11): "도입 안 함 확정"도 틀렸다 — 세 번째 반전.**
    - 위 "실물 검증"이 사실은 **엉뚱한 이미지**를 본 것이었다. 내가 스크린샷한 건 `.cont_detail`의 **상품 사진**(`_detail_NNN.jpg`)이었고, 정작 **텍스트가 든 상세 composite(`_DC.jpg`)는 다른 컨테이너(`#detail`)**에 있었다. 즉 "검증했다"면서 또 일부만 보고 단정한 것.
    - 사용자가 **실제 `_DC.jpg`(800×8756, 설명·COLOR·SIZE·FABRIC·스펙표 가득)**를 직접 제시 → happyland 상세엔 텍스트가 **분명히 있었다**.
    - **추가 교훈**: 통짜 OCR은 다운스케일로 환각(유아 사이즈 80/90/100/110 → "S/M/L", 면87%→"면100%"). → **세로 스트립 분할 + 512px 리사이즈**로 정확도 확보, **동시성 제한+429 재시도**로 TPM 보호.
    - **최종 구현**: 조건부 OCR — `셀러태그·본문 없음`(근거 부족)일 때만 상세 이미지 OCR로 detailText 확보 → USP/hashtags를 실제 소재·특징 기반으로 생성, provenance에 `상세이미지OCR` 정직 표기. 네이버(셀러태그 보유)는 스킵 → 비용 0.
    - **메타 교훈**: 같은 실수(일부만 보고 과일반화)를 **세 번** 반복했다(#6 본문, 후속 "검증", 그리고 이 정정). "검증했다"는 말 자체도 검증 대상이다 — 무엇을 봤는지가 핵심.
- USP 과장·미입증 효능(질병 치료 등)은 프롬프트로 억제했으나, 의약품/건강기능식품 표현 규제까지 완벽 검증은 못 함.

## 7. lowest_price(가산점) — 시장 최저가 매칭: 오탐 ↔ 과탈락 줄타기 ★

**목표**: lowest_price = 상품 페이지 할인가가 아니라 "시장 전체 실제 최저가". 명세가 **"오탐 방지가 핵심"**이라 못 박음 → 틀린 값 채우느니 null.

**소스 선택 (쿠팡 우회)**
- 쿠팡 직접크롤(Akamai 차단)·파트너스 API(키 요건)가 막혀, **에누리(가격비교)를 합법 우회로**로 사용 — 쿠팡 포함 오픈마켓(11번가·G마켓·옥션·SSG 등) 가격을 한 번에.
- 네이버쇼핑 OpenAPI는 **syncNvMid(productId) 정확매칭** 가능 → 가장 강한 신호.
- 에누리는 가격이 JS 렌더라 Playwright 필요(고도몰의 순수 fetch와 대비). 가격·판매처는 **DOM 텍스트 직접 추출**(OCR 아님): `li.prodItem` → `.col--price em`(가격)·`.price__mall img[alt]`(판매처)·`.ico-tag--coupon`(쿠폰).

**핵심 난제 = 동일상품 매칭. 두 번 실패하며 배움:**

1. **false positive(틀린 최저가) ×2**
   - **묶음 vs 단품**: 우리 "1+1 케피 포인원"(25,300)에 단품(12,930 ≈ ÷2)을 매칭. → **N+M 묶음 배수 가드**(bundleQty).
   - **여행용 vs 정품**: "케피 버블클렌저 3개" 최저가순 1위가 **여행용 3종 트래블키트**(12,720). 토큰(케피+버블클렌저)·단위(3종=3개)가 다 같아 통과. 처음엔 **키워드 리스트(여행용·트래블…)**로 막으려다 "두더지잡기(다음 변종은 또 샘)"라 **폐기**.

2. **약한 신호의 한계 인정**: "브랜드+카테고리 토큰"만 공유하는 상품군(케피 버블클렌저)은 변종이 구조적으로 새어든다. → **신뢰 신호 계층화**
   - 강(결정적): **네이버 pid==mid** / **모델코드 일치**(H1321210·A131300 — 몰 무관 동일상품 식별).
   - 약(모호): 코드·mid 없음 → **LLM이 "동일상품?" 의미 판정**(어려운 20%만). 변종을 키워드 없이 의미로 걸러냄.

3. **false negative(과탈락)**: LLM-only로 갔더니 **코드가 명백한 코벤트·도티(정탐)를 33% 가격차에 과보수적으로 거부**. → 내 설계 실수(결정적으로 풀 코드매칭을 LLM에 떠넘김). **모델코드 일치=strong(LLM 생략)**으로 교정 → 코벤트(23,520)·도티(32,720) 복구, 여행용은 LLM이 계속 컷.

**최종 구조 (3층 방어)**: 결정적 가드(가격sanity·브랜드·토큰·단위·묶음) → 강한 신호(mid·코드)는 확정 → 모호한 후보만 LLM → 통과 없으면 null+사유. provenance에 경로(`pid==mid`/`모델코드확정`/`LLM확인`)를 정직 표기.

**쿠폰 정책(③)**: 에누리 쿠폰 적용가는 **조건부(클립·기간한정)라 신뢰도 낮음** → lowest_price엔 **비쿠폰 표시가만** 사용(쿠폰행 제외). "누구나 사는 가격"에 집중.

**시장최저 > 판매가(④)**: 케피 포인원처럼 타몰 최저(27,800)가 판매가(25,300)보다 비싸면 → **둘 중 더 낮은(=사실) 판매가**를 채우고 "타몰 최저 X" 병기. 공란 대신 사실을 기록(판매처가 최저라는 것도 제안서에 가치).

**가장 솔직한 회고 — "검증했다"는 말의 재검증 (#6 메타교훈 재발)**
- 테스트 스크립트가 `LLM 검수: ON`이라 출력했지만, 실은 **셸의 죽은 OPENAI 키로 401 → catch로 조용히 폴백**하고 있었다. 강한 신호(mid·코드)가 결과를 받쳐줘 **틀린 줄도 몰랐다.** (main.ts는 `dotenv override:true`로 .env 우선, 스크립트는 안 해서 셸 변수에 가림.)
- **단일 상품 트레이스**를 떠 LLM 호출까지 실제로 찍어보고서야(여행용 → "용량·구성 다름" 판정) 진짜 동작을 확인. 자동 폴백은 편하지만 **실패를 삼켜 거짓 안심**을 준다 → #6의 "무엇을 봤는지가 핵심"을 또 확인.

**해결 못한 것 / 더 시간이 있었다면**
- 모델코드 추출은 정규식(`[A-Z]{1,3}\d{4,}`) 휴리스틱 → 코드 없는/포맷 다른 상품은 LLM 의존.
- LLM 판정 비용·지연 → 동시성 4 제한 + 강한 신호는 LLM 생략으로 호출 최소화. 전수 규모(59+) 정밀도는 전체 크롤로 추가 검증 필요(샘플 5는 오탐 0·과탈락 0 통과).

## 8. 가산점 — 가설을 데이터로 검증해 재설계 ★ (자가복구·증분·옵션 LLM)

가산점 기능을 붙이며 가장 중요했던 건 "이 LLM이 정말 밥값을 하나?"를 **가정하지 않고 실데이터로 검증**한 것. #6의 교훈("검증했다는 말 자체도 검증 대상")을 그대로 반복 적용했다.

**옵션 의미배치 — "LLM이 도움 된다"는 가정의 붕괴.**
- 다축 옵션을 LLM이 큐닷 2칸(option1=종류, option2=구성)으로 재배치하게 했었다. 그럴듯해서 의심 없이 썼다.
- "꼭 필요한가?"를 묻고 **룰(위치) vs LLM 비교 하니스**(`npm run compare:options`)를 실데이터(kefii 378 SKU)로 돌리니 — LLM이 ≤2축에서 **상품명 누출**(`"케피 바디워시 애플포레스트"`)·**색상 뭉침**·`null` 조각 같은 **오염을 추가**하고 있었다. 룰은 깨끗했다.
- 또 하나의 발견: 네이버 축 이름(groupName)이 `"버블클렌저 3개 선택 × ★무료증정★"`처럼 **판매자 노이즈**라, "맛→option1" 식 결정적 의미배치는 불가능. 현실적 선택지는 위치(깨끗하나 단순) vs LLM(똑똑하나 오염)뿐이었다.
- 재설계: **≤2축은 룰(위치)·LLM 미사용**, **3축만 LLM**. 3축 오염은 self-heal과 같은 **grounded·무손실 가드**로 차단 — LLM 출력의 모든 토큰이 입력 값에 실재하고(무오염) 입력 값이 모두 담겨야(무손실) 채택, 아니면 위치 폴백. 결과는 항상 "깨끗한 LLM 또는 깨끗한 위치값", 절대 안 깨짐. (검증 `npm run test:options` 10 pass)
- 교훈: LLM은 "기본"이 아니라 "검증된 어려운 20%"에만. 가정을 데이터로 깨고 **범위를 좁히는 것**이 기능을 더하는 것보다 가치 있었다.

**자가복구(self-heal) — LLM을 안전망으로.**
- 결정적 추출(셀렉터/JSON)이 핵심 필드(name·정가)를 비우면, 어댑터가 보존한 원본을 LLM이 재분석해 복구. 단 **grounded**(원본에 실재하는 값만)·`ai-recovery` 정직 표기·평소 호출 0(빌 때만). "지어내지 않는다" 원칙을 LLM 복구에도 강제.
- 옵션 복구는 **의도적 보류**: 트리거 모호(빈 옵션이 "없음"인지 "추출 실패"인지) + LLM이 옵션 추가금을 만들면 "가격은 LLM에 안 맡긴다" 원칙 충돌.

**증분 재크롤.**
- 목록+배치가격으로 이전 결과와 diff → 신규/가격변경만 재크롤, 나머지 재사용. 비싼 작업(에누리 브라우저·OCR·자가복구) 절약 + 호출 횟수↓(크롤 매너).

세 기능 모두 라이브 크롤 없이 검증 가능한 하니스로 못박았다(`npm run verify`/`test` — 합성 hard 케이스를 실제 코드에 통과).

## 한 줄 요약 (회고 총평용)
> "많이/빠르게 긁기"가 아니라 "바로 쓸 수 있는, 근거가 검증된 데이터"에 집중했다.
> AI는 비싸고 틀릴 수 있는 부품으로 다뤘고(룰 우선·검증·fallback),
> 못 채운 값은 지어내지 않고 사유와 함께 비웠다(provenance + basis 투명 공개).
> 가장 큰 교훈: "상세 본문 텍스트 추출"처럼 그럴듯한 기능도 실제 데이터로 검증하니 허상이었다.
> 기능을 남기기보다 검증으로 허상을 걷어내고 근거를 정직하게 표기한 과정 자체가 이 과제의 핵심이었다.
