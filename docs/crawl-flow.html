1인증 세션 오픈
BrowserSession.open() — persistent 컨텍스트로 사용자 1회 로그인 세션 재사용 + stealth.
curl/직접 fetch는 429·캡차 → 실브라우저+세션이 정공법
2채널 식별
스토어 진입 → __PRELOADED_STATE__에서 channelUid·channelNo, API prefix(/i vs /n) 결정.
3목록 전수 페이지네이션
1페이지: SSR categoryProducts(40개). 2~N페이지: 스크롤 → 페이지 버튼 클릭 → XHR 가로채기.
목록 직접 fetch는 서명헤더 부재로 429 → 클릭+가로채기
4가격 배치 조회
POST product-benefits (50개씩 청크) → 정가·즉시할인가·배송비. 개별 호출 회피.
성능 규칙: 가격은 배치로
5상세 추출 (상품마다)
GET products/{id}?withWindow=false → 이름·이미지·옵션·wholeCategoryName·sellerTags·syncNvMid. 배치 가격 병합.

실제 크롤링 결과 예시

각 스토어에서 추출·정규화한 큐닷 상품 제안서 — 사이트당 2건 · 전체 JSON에는 필드별 provenance(근거)·meta 포함
설계 원칙 (타협 금지)
  • 결정적 추출 — 화면 셀렉터가 아니라 내부 JSON API / data-* 속성에서 추출
  • 룰 우선, LLM은 어려운 20%만 — 가격·이미지·이름은 코드로, 분류·요약만 LLM
  • AI 출력은 신뢰하지 않는다 — 모든 LLM 출력은 validate 관문 통과
  • 지어내지 않는다 — 못 얻는 값은 공란(null) + 사유(provenance)
  • 크롤링 매너 — 요청 딜레이 유지, 본인 인증 세션 재사용, 차단 우회 자랑 금지
